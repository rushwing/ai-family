07 — 源码仓库与代码治理策略

按 GCP Enterprise Foundations / Vertex Agent Starter Pack 与 Azure CAF / azd 对「SMB AI 方案」的风格，给出 Client 通道 / Agent / Toolset-MCP / Data / Security-Governance 五层的源码组织、仓库边界、CI/CD 与分支治理。本页是 ADR-013（待立项）的设计依据；衔接 ADR-009（compose→K3s）与 05 里程碑。

1. 问题与约束

需要决定的是：五层源码用一个仓还是多个仓、边界划在哪、各仓的 CI/CD 与权限怎么配。约束（事实/硬约束）决定了答案的形状：

📘 约束（事实）：① 单人开发（rushwing），无团队边界红利可言；② 五层高度耦合——Agent 共享 LangGraph 基类、MCP 工具契约、checkpointer 状态 schema、JWT/RLS 鉴权契约；③ 既有 harness（REQ/TC/BUG）+ GLOSSARY 是单一权威，跨层引用频繁；④ 既有资产 rushwing/goal-agent（带历史）需并入；⑤ Phase 1 docker-compose、Phase 2 K3s（GitOps 友好，ADR-009）。
🚫 硬约束（kid 安全红线）：内容/合规策略「在引擎不在 prompt」（04 P6 / 03 §4）。谁能改 kid 安全策略、RLS、审计 schema，必须是比"改一个旅行 Agent"更小、更受控的圈子——这是仓库边界划分的第一性原则，而非可选项。

2. 业界参照：GCP / Azure 对 SMB AI 的仓库风格

两家公有云对「中小企业级 AI 方案」的官方风格高度一致：应用用模块化单体仓，按「安全 + 生命周期边界」拆出极少数仓，绝不按每个微服务/Agent 拆仓。Per-service 多仓只在组织级、多团队（康威定律）才出现。

✅ 为什么 B（每 Agent 一仓）在此规模不成立：scalable 的真正瓶颈不是仓数量，而是共享契约的变更成本。Agent 共享基类/工具信封/状态 schema，把它们切到多仓只会把"一次提交"变成"跨仓协调"。等真正出现独立团队或要单独开源某 Agent 时，再用 subtree split 把那个模块切出去——届时收益 > 成本，缝也早已预留。

4. 推荐：模块化单体仓 + 两个边界仓（三仓）

✅ 决策（经 2026-06-14 裁决修订）：采用 C 的单仓变体 —— 初期纯 Monorepo，三层逻辑边界靠 path filtering + CODEOWNERS，不第一天分物理仓：
· 现在：单仓 ai-family（应用全层 + 共享库 + harness + docs），infra/（IaC/GitOps）与受 CODEOWNERS 双签保护的 governance/ 均为目录。
· 后续按触发条件拆物理仓：Agent 规模 / 团队 / 合规审计需要时，再把 infra/、governance/ 或某 Agent 模块 git subtree split 为独立仓（kid 上线 M3 前优先评估 governance）。
Agent 始终模块化但不拆仓。

为什么不第一天分仓：单人开发时，跨层改一处（如加一个环境变量）若要在两个物理仓各提一个 PR，会拖慢 Draft-First 的迭代节奏；而仓内 infra/、governance/ 目录 + 严格 CODEOWNERS + path filtering 已能拿到部署解耦与"小圈子强审"。真正的物理隔离（blast radius / 合规职责分离）留到 Agent 规模或审计需求出现时，再沿目录边界 git subtree split，迁移成本低。这是 Gemini（反对分仓）与 Codex（主张分仓）分歧的裁决落点：先迭代速度，后物理隔离。

│  ├─ state-schema/             #   checkpointer 状态模型

├─ .github/
│  ├─ CODEOWNERS                # 模块 owner；governance/ 路径强制 human-001
│  └─ workflows/                # path-filtered CI：改 agents/goal/** 只构 goal-agent
└─ compose.dev.yaml            # 本地起栈（生产声明在 infra/ 目录）
📘 关键纪律：跨层共享的东西一律进 libs/，且只进 libs/。Agent 之间不得横向 import 彼此的 src/，只能依赖 libs/ 的契约——这条规则让"同仓"不等于"耦合成一坨"，也让未来 subtree split 单个 Agent 时依赖关系清晰可断。

7. 分支与代码治理

  • Trunk-based + 短命特性分支：主干常绿，特性分支命名带 REQ-ID，PR 标题/commit 关联 REQ/BUG（承 reference 的 harness 纪律）；不搞长命发布分支（家庭规模无并行版本需求）。
  • CODEOWNERS 分层：agents/* 模块各自 owner（单人阶段都是自己，但保留结构）；libs/**（共享契约）改动需更谨慎的自审清单；ai-family-governance 全仓 / 或平台仓 governance/ 路径 = human-001 强制双签。
  • 多 AI 评审入流水线：单人项目里"人审"形同虚设，由 Codex/Gemini 多视角评审补偿（05 §3）；评审意见按 harness 回流为 BUG/REQ/ADR Review Notes，不直接改稿。
  • 保护分支：main 禁止直推；CI 绿 + 评审通过方可合并。
🚫 kid 红线的代码治理映射：任何触及 kid 安全策略 / Compliance 规则 / RLS / 审计 schema 的改动，必须落在受 CODEOWNERS 双签保护的 governance 边界内，且策略以版本化 bundle 只读下发给 Agent/Compliance 消费——Agent 仓的任何改动都无法绕过或反向篡改策略。这把"策略在引擎不在 prompt"从运行时一路保障到源码权限层。

8. 落地步骤与 ADR-013 衔接

  1. M1 起即落骨架：建 ai-family（已是当前仓）的 apps/ agents/ toolsets/ data/ libs/ 模块骨架 + path-filtered CI；infra/ 与 governance/ 作为目录起步（不第一天分物理仓，裁决 2026-06-14）。
  2. 并入既有资产：rushwing/goal-agent 用 git subtree add 导入 agents/goal/（保留提交历史），随 GoalAgent 垂直切片（06）改造（MariaDB→PG、header→OIDC、APScheduler→图唤醒）。
  3. governance 先目录后独立：单人初期可先用平台仓 governance/ 目录 + CODEOWNERS；kid 上线（M3）前升格为独立仓 ai-family-governance。
  4. M5 K3s：infra/ 的 compose 声明 Helm 化，转 GitOps（ADR-009 混合编排：仅 Pi5 入 K3s，Mac/NAS 外部接入）。
  5. 立 ADR-013：本页结论经 Codex/Gemini 评审后，落 ADR-013 源码仓库与代码治理（proposed→accepted），与 ADR-001/009 交叉引用。

9. 风险与反悔条件

编号风险对策 / 反悔条件
RR1monorepo 全量 CI 变慢path filter + 依赖图增量构建 + 缓存；仍慢则按模块拆 CI workflow（不拆仓）

RR3某 Agent 真需独立团队/独立开源/独立发布节奏触发 git subtree split 把该模块切出为独立仓——边界已预留，迁移成本低

RR5libs 变"上帝包"，模块隐性耦合libs 按契约细分子包；Agent 间禁止横向 import（§5 纪律）；定期依赖审查
✅ 一句话总结：monorepo now, polyrepo-ready —— 用模块边界拿到 polyrepo 90% 的好处（独立构建/发版/所有权），用单仓避免它 90% 的成本（跨仓契约对齐）；只在 blast radius 与 kid 红线这两条真实边界上才拆出独立仓。
