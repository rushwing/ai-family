面向家庭多成员的 production 级多智能体平台。参考 GCP 企业 AI Agent 解决方案方法论（reference.html），以 GoalAgent 垂直切片为起点，以点到面。本页是 L1 总览，读完即可决定深入哪份 L2 文档。

1. 问题域（Problem Scope）

为一个家庭（约 4–6 名成员，含未成年人）提供一组各司其职的 AI 智能体：目标与日程管理（GoalAgent）、学习辅导（StudyAgent）、旅行规划（TravelAgent）、投资研究（StockAgent）、深度调研（ResearchAgent）、家庭知识库（KnowledgeAgent）、家庭数据访问（DataAgent），由意图路由与规划器编排，受合规审查与全链路审计治理。

同时，本项目是一次刻意的工程练习：用企业级 AI 落地的标准流程（设计 → 评审 → 垂直切片 → 灰度放量 → 平台沉淀）走完一个真实项目。因此选型一律取 production 级组件，按"小微企业规模"取中，不考虑高并发。

📘 设计输入（事实 · 2026-06-13 实测修正）：硬件 = 树莓派5（4GB · 512G SSD · 轻量接入节点）、绿联 NAS DX4600（Intel N5105 · 16GB · 数据层）、Mac Mini M4（16GB+256GB · 618 入手 ~¥3000 · 主运行时 + OpenClaw 宿主）、公网域名、可小额租 VPS。既有资产 = goal-agent（FastAPI+FastMCP，36 tools）、travel_agent（MCP/ReAct 教学实现）、OpenClaw skills（tutor / kids-coding 等）。技能栈 = React/NextJS/NodeJS、FastAPI、gRPC、RabbitMQ/Kafka、Docker/K8S、PostgreSQL。

2. 设计原则（每条都会被后续决策引用）

#原则含义与边界
P1Production 级，不用玩具每个组件都是企业真实会用的（LangGraph/PG/Keycloak/RabbitMQ/MinIO…）；但容量按家庭取最小形态，不为不存在的并发设计
P2垂直切片优先不平铺全量建设。GoalAgent 先全链路打通（reference 的 FDE 答案："先建垂直 Agent，再抽平台能力"），其余 Agent 保持框图深度
P3身份链端到端Agent 读到上下文 ≠ 用户有权限。鉴权在工具侧 + DB RLS 兜底，身份从 CF Access 一路透传到数据行
P4数据不出家门对话、文档、审计数据自托管；只有发给 LLM API 的请求内容出网（这是接受的边界）；重要数据加密异地备份
P5Draft-First 写操作一切写动作先草稿后确认，确认权只在人手里；含未成年用户的系统不做"全自动执行"
P6规则不进 Prompt儿童策略、工具白名单、配额放策略引擎/网关配置，prompt 只装上下文（承自 reference 的警告）
P7Scope 由前端设计稿锚定Claude Design 产出的 UI 稿 = 产品边界，设计稿之外的功能不进开发队列
P8决策留痕可追溯选型必有 ADR（含落选项与重审触发线）；变更先改文档再改实现；术语以 GLOSSARY.md 为准
