3. Planner 的 Plan Mode（对标 Claude Code plan mode）

  1. 触发条件：IntentRouter 判定为复杂任务（跨 Agent / 预估多步 / 含 write 风险）——简单读类请求不进 Planner，直达领域 Agent（省 token、降延迟）。
  2. 计划生成：Planner 产出结构化计划（步骤、负责 Agent、依赖边、预估成本、风险标记），以人类可读卡片渲染到 ChatUI。
  3. 批准门：LangGraph interrupt() 挂起图执行，等待用户【批准 / 修改 / 拒绝】。状态落 checkpoint——用户睡一觉再批，图原地恢复。
  4. 执行：按 DAG 并行调度子图；每节点失败重试 ≤2 次后降级（部分结果 + 失败说明，绝不编造缺失结果——承自 reference 的 fallback 原则）。
  5. 写动作二次门：计划批准 ≠ 写动作批准。每个 write 工具仍走 Draft-First 两段式（ADR-010），Planner 无权自我确认。
⛔ 硬约束：Planner 不能确认自己的写操作。确认 token 只能由 ChatUI 的用户交互产生，经网关校验来源。这是含未成年成员系统的不可协商项（原则 P5）。

4. 上下文装配配方（Context Engineering）

prompt_context =
  system_policy        # 该 Agent 的角色与边界（静态，prompt cache 友好，放最前）
+ user_profile         # 来自 JWT + 成员画像（role、年级、偏好）——不含敏感明细
+ task_intent          # IntentRouter 产物（意图 + 风险等级）
+ allowed_tools        # MCP 网关按角色下发的白名单（kid 看不到的工具不存在于上下文）
+ retrieved_evidence   # KnowledgeAgent 混合检索 + 图谱一跳（带来源 id，输出须引用）
+ conversation_state   # LangGraph State（checkpoint 恢复后完整可用）
+ prior_tool_results   # 本轮已执行工具的结构化结果
+ output_schema        # 结构化输出契约（UI 渲染 + 审计解析）
  • 顺序即缓存策略：静态段（system_policy）置前，命中各厂商 prompt cache；动态段置后。
  • 规则不进 prompt（原则 P6）：儿童限制、配额、工具白名单在策略引擎/网关判定，prompt 只接收判定结果。
  • 检索内容是数据不是指令：retrieved_evidence 与 untrusted 工具输出以明确定界符包裹，Compliance 输入侧对其做注入扫描（03 §4）。
