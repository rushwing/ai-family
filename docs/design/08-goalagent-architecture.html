08 — GoalAgent 重审后架构（产品 scope + 平台化调和）

对 rushwing/goal-agent 做一次 产品 scope 重审 + 架构重设计——不是 MariaDB→PG 的组件替换，而是结合 ai-family 顶层设计（01）把它收敛为平台化垂直切片。本页是 GoalAgent 的 L2 架构详设；06 保留为现状盘点/迁移清单。对应 REQ-003。

三色笔记语义：绿 = 决策/结论橙 = 取舍/权衡红 = 风险/硬约束

1. 重审目的与基线

goal-agent 是一个已上线、功能完整的"目标与习惯跟踪 Agent"（FastAPI + FastMCP 36 tools + MariaDB + Telegram + APScheduler，单 coach-多 pursuer 角色模型）。把它接入 ai-family，最低成本是"换组件"；但那样会把 goal-agent 的单用户假设、外部数据通道（GitHub/Telegram）、未成年人无关的产品形态原样带进一个含 kid 的多成员平台。本稿因此先做产品 scope 重审，再做架构调和。

基线来源内容对本稿的作用
goal-agent 现状（8 阶段）核心闭环 + 周/月报 + GoalGroup 向导（已部分 LangGraph 化：wizard_graph.py）+ web_research/feasibility + Gamification+（未建）+ OpenClaw 插件 + GitHub 归档 + Telegramscope 重审的输入
goal-agent 自带迁移计划docs/langgraph-migration/：Strangler Fig 五阶段，仅聚焦 wizard_service 子图的组件级 LangGraph 化（langchain-anthropic + AsyncSqliteSaver）可复用的局部资产，但层级不够
ai-family 顶层设计01（六层架构 / IntentRouter / Planner / Draft-First / MCP 网关 / PG checkpointer）、06（垂直切片盘点）、03（RLS 租户 + Compliance）调和目标
🟠 核心张力：goal-agent 自带计划是"把 wizard 的控制流换成 StateGraph"（组件级、SQLite checkpointer、单用户）；ai-family 要的是"目标全生命周期长程图 + 多成员 RLS + OIDC/网关/审计/LiteLLM"（平台级）。本稿的取舍主线，就是把 goal-agent 的局部 LangGraph 资产收编进平台长程图体系，而非沿用它的迁移层级。

3. 架构调和：组件迁移 → 平台化运行时

goal-agent 原架构是"FastAPI 进程内挂 FastMCP + 进程内 APScheduler + header 鉴权 + 直连 MariaDB"，单进程、单用户、信任边界模糊。重审后它成为 ai-family agent-core 内的一个领域 Agent 子图，所有数据/工具访问被收进平台的信任边界与租户隔离：

📘 层级修正：goal-agent 自带迁移计划停在"wizard 控制流换 StateGraph"。本稿把它上提一层——wizard 只是 GoalAgent 名下的一个子图，与主长程图共享 PG checkpointer、同受 OIDC/网关/RLS/审计约束。goal-agent 的 SQLite checkpointer、langchain-anthropic 直连均不沿用（统一走 PG checkpointer + LiteLLM）。
🔴 M1 kid 路径 = 结构化只路径（BUG-012）：Compliance Agent 是 M3，而 01/03/GLOSSARY 要求 kid I/O 必过 Compliance。故 M1 不让 kid 自由对话 GoalAgent：kid 仅 ① RLS 下只读自有目标 ② Draft-First 打卡（自有任务）③ praise_engine 离线模板赞语（不走 LLM 自由生成）。kid 不可创建/批准平台或他人目标写入，越界请求温和转家长。如此 M1 无未审 LLM 输出给 kid → 不需 Compliance 亦满足 P5 红线。kid 自由对话 + 完整 Compliance v0（输入注入/意图过滤 + 输出 kid 100% judge + PII 脱敏 + compliance_block 审计）顺延 M3。

状态权威矩阵与节点事务契约（BUG-013）

goal-agent 上游迁移稿（02-comparison）以业务表为 source of truth、checkpointer 只管控制流。本稿沿用并明确三者权威与恢复规则：

存储记录什么恢复时的角色
DB 业务表（goals/plans/checkins…）业务真相（目标/计划/打卡/XP）🟢 恢复权威（reconcile 基准）
outbox（同库表）待投递副作用（通知/报表）+ idempotency_key重放去重，至少一次投递
LangGraph checkpoint控制进度（走到哪个节点）仅恢复控制流，非业务真相
🔴 节点事务契约：副作用节点的业务写入 + outbox 投递 + idempotency_key 落库必须在同一 DB 事务；checkpoint 在事务外另写。恢复规则：以 DB 业务状态为权威 reconcile checkpoint/outbox——若 commit 成功但 checkpoint 失败，重放时凭 idempotency_key 去重、不重复副作用；若 checkpoint 领先于 DB（业务事务回滚），以 DB 为准重做该节点。与 BUG-001 同源。

5. 安全 / 正确性内建（把缺陷升为一等需求，非"兜底验收"）

原 REQ-003 把 BUG-001/002 列在"兜底验收"。本稿据 review 裁决将其上提为设计内建需求——在迁移与图设计阶段就落地，因为这些是 kid 安全与数据正确性的根，事后回填代价极大。

🔴 BUG-002（RLS critical · kid 红线）：随数据层迁移步骤一次性内建——① 所有业务表 FORCE ROW LEVEL SECURITY；② 迁移/DDL 走受限 role，应用 role 无 BYPASSRLS；③ 后台 worker / 图节点写库必设 member/shared scope（不得以超级身份跑）；④ 默认禁 SECURITY DEFINER 函数，例外须评审；⑤ 若用 PgBouncer transaction 模式，必须事务内 SET LOCAL per-txn claim（禁 session 级，防连接复用串号）；⑥ CI 扫描：新增表无 RLS 策略即失败。详见 03 与 ADR-003。
🔴 BUG-001（精确一次语义）：LangGraph checkpoint ≠ 精确一次。副作用节点（提醒、计分、报表、写库）须以 idempotency_key 去重 + outbox 投递，kill-retry 不产生重复工具执行或重复通知（见 §4 图 + ADR-001）。
🔴 BUG-011 + BUG-015（审计 append-only 落库约束）：审计须与可删会话历史物理分离——独立 audit schema、专用 INSERT-only DB role、用 trigger/policy 禁 UPDATE/DELETE，必要时 hash chain / 只读副本。kid 删聊天、软删除、app 角色改写、后台误清理均不得触及审计。与 REQ-008（被遗忘权）的 retention 张力：被遗忘权作用于业务/会话数据，审计保留最小必要元数据（可加密/假名化），二者分层（见 ADR-012 / ADR-015）。
🔴 BUG-014（RLS 覆盖平台状态表，不止业务表）：所有 tenant-bearing 表都必须有 family_member_id/shared scope + RLS + FORCE RLS（或更严格权限）。除业务表外，平台状态表清单同样纳入 CI RLS 扫描：
平台表含租户/权限数据策略
langgraph checkpoint会话/计划中间态（可含 kid 数据）RLS by thread→member（03 §3 已要求）
outbox待投递副作用（含成员目标内容）RLS + member scope
audit谁对谁做了什么INSERT-only role + 读侧 RLS
confirm_token写操作确认令牌（绑成员/来源）RLS + 短 TTL + 来源校验
tool registry / call log工具调用记录（含参数）RLS + member scope
BUG-002 的加固范围据此扩展到上述平台表。
📘 这些与原 REQ-003 的关系：仍由 REQ-003 的实现承载（linked_req=REQ-003），但定位从"验收末端检查"改为"迁移①/图③/审计层的设计输入"。REQ-003 差距清单据此刷新。
