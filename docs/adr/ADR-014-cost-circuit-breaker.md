## Context（背景与约束）

Gemini 外部评审指出 M0 设计的【最大单一缺口】：完全没有“LangGraph 陷入 ReAct 死循环（不断调用工具失败再重试）如何防止一晚刷爆 API 账单”的决策。在家庭预算下，失控成本比系统宕机更致命——这是必须前置立项的硬约束。
现有 ADR-002（LiteLLM）只有“月度预算硬顶”这一粗粒度护栏，无法阻断单会话级的瞬时失控，也不管引擎侧的循环步数。

## Options

### Option A: 仅靠 LiteLLM 月度预算硬顶（现状，否决）
**Pros:** 零新增。
**Cons:** 月顶是事后粗粒度，单晚死循环可在月顶触发前刷掉大额；不阻断 ReAct 循环本身；无会话/成员级粒度。

### Option B: 网关层 + 引擎层双层熔断（选定）
**Pros:**
- 网关层（LiteLLM）：按会话/日/成员的 token 与金额阈值，超限即拒绝后续调用（fail-closed）——金额护栏的权威落点。
- 引擎层（LangGraph）：ReAct 循环步数上限 + 工具连续失败重试上限，超限即终止图并告警——阻断“失败-重试”死循环的根因。
- 双层互补：金额护栏防总量、步数护栏防循环；任一触发都能止损。
**Cons:** 需在两处实现并联调阈值；阈值过紧会误杀正常长任务（用可配置 + 告警而非静默对冲）。

### Option C: 仅引擎层步数上限
**Pros:** 改动小。
**Cons:** 挡得住循环，挡不住“单次超长上下文/高档模型”的金额失控；无成员级配额。

## Decision

采用 **Option B**：在 **LiteLLM 网关层**实现基于 token/金额的硬熔断（会话级 / 日级 / 成员级阈值，超限 fail-closed），在 **LangGraph 引擎层**实现 ReAct 循环步数上限与工具连续失败重试上限（超限终止图 + 告警）。阈值可配置，超限事件写审计（ADR-012）并经 `notify.out` 通知。实现见 **REQ-007**。

## Trade-off

接受双层实现与阈值调参成本，换取“家庭预算下不会被死循环刷爆账单”的硬保证。先做保底硬熔断，**放弃**精细的动态预算优化（按价值/优先级动态分配额度）留待后续。阈值偏保守可能误杀极少数正常长任务——以可配置 + 告警可见（而非静默丢弃）对冲。

## Consequences

- ADR-002 的 LiteLLM 从“月度硬顶”扩展为多粒度（会话/日/成员）熔断点；预算超限 TC 必须打真实网关验证（与 BUG-003 协同）。
- LangGraph 图模板统一注入步数/重试上限节点（编排底座 ADR-001 的横切关注点）。
- 熔断触发是可观测的一类事件：进审计表 + Langfuse + 通知，便于复盘阈值。

## Revisit Trigger

- 误杀正常长任务 ≥ 2 次/月（重调阈值或引入任务级豁免白名单）。
- 出现需要按价值动态分配额度的场景（评估从硬熔断升级为动态预算）。
