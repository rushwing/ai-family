## 现象

REQ-003 的 TC 代码虽能静态编译和在缺环境时 skip，但多处无法对承诺的安全/正确性属性给出有效证据：

1. TC-003-01 在数据库中创建无 RLS 表和 SECURITY DEFINER 函数，却调用只扫描 data/migrations 文件的 tools/check_rls.py；扫描输入未变，正确的扫描器仍会返回 0，负例会稳定误失败。该用例还把审计平台表写为 public.audit_log，而 TC-003-06 要求独立 audit.event schema。
2. TC-003-02/04 的网关认证路径引用未实现的 tests.helpers.oidc，并把 login_code_pkce 拼成 login_code_pkke；即便环境齐备，涉及真实身份的断言仍会 skip 或导入失败。
3. TC-003-09 没有用 master key 创建带确定性小额度的测试虚拟 key，却循环尝试最多 50 次真实 completion，只验证 deepseek；运行脚本也没有要求 fail-closed 用例所需的 AIFAMILY_BASE_URL。
4. TC-003-10 先写入固定文字“REQ-002 done; REQ-003 tc_impl”再检查报告含这两个字符串，未证明报告来自仓库当前 source，也没有验证全量 REQ 重导不重复。
5. TC-003-04 只验证注册总数为 36，未固定逐 tool 名称/风险/角色/scope 契约，可由 36 个错误或空语义 tool 假绿。

## 期望行为

每个 TC 在目标组件和受控依赖就绪后应当实际验证其宣称的安全/功能属性；环境未就绪时可以 skip，但不得把缺失测试支撑、错误扫描对象、伪造输入或不可控计费伪装成通过证据。

## 根因分析

测试先行的 env gate 已建立，但部分用例把将来接口名、测试输入和断言关系假定为存在，未校验它们是否与被测系统、扫描器边界及真实凭证操作一致。

## 修复方案

1. 令 TC-003-01 的负例向迁移扫描器可配置的临时迁移目录注入 SQL（或将扫描逻辑抽成可传目录的函数），并统一 audit 独立 schema 的表名/迁移契约。
2. 实现并纳入测试依赖的 code+PKCE 登录 helper，统一命名为 login_code_pkce；认证依赖缺失应使完整环境用例失败而不是静默 skip。
3. 以 master key 在隔离租户创建并清理两个小额度虚拟 key，分别断言三家真实 provider completion、独立计量和明确 429；运行脚本补齐 AIFAMILY_BASE_URL，限制请求数并输出脱敏证据。
4. 令 TC-003-10 从版本化 harness source 读取实际 M0–M5 与 REQ 状态，比较导入/周报内容与 source；重导后同时比较 milestone 和 requirement 的去重结果。
5. 在 TC-003-04 固定 36 个预期 tool 的名称与 manifest 契约，并覆盖误标为 read 的写工具和确认 token 的用户来源绑定。

3. 以隔离 key 的实际已计成本设置或收紧 max_budget（或使用可查询的 budget API）后再发一次确定请求，保证有限次数内确证 429，并保持三 provider 成功与 A/B 独立性断言。

## 重新打开（round-3）

- [evaluator-001][2026-06-20] round-2 的 manifest 角色与预算校准修复有效，但 TC-003-01 仍无法在真实源 schema 上执行：

3. 修复后由 human-001 以 T17 将 REQ-003 从 blocked 恢复 tc_impl/generator-001，再重交 T09。
