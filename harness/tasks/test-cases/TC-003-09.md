## 前置条件

- agent-core 仅连接 compose 的 agent_internal 网络，并通过 litellm:4000 调用模型。
- 已提供受控的真实厂商凭证、LiteLLM master key、两名成员的虚拟 key 与可清理的测试预算；运行日志脱敏。
- 用例在隔离测试租户执行，不在 CI 或提交日志中写入真实凭证。

## 步骤

1. 从 agent-core 容器尝试解析/连接厂商域名，并连接 litellm:4000。
2. 临时使 LiteLLM 不可达后调用 Agent LLM 功能，监测错误、网络连接和任何 provider fallback。
3. 使用两个不同成员的虚拟 key，分别执行真实 completion 至额度边界，再尝试超额请求。
4. 用缺失或错误的成员/虚拟 key 调用；审查网关计量与 Agent 审计记录。

## 期望结果

- agent-core 无法直接出网到厂商，仅能经 litellm:4000；不存在裸调厂商的成功路径。
- LiteLLM 不可用时 Agent 明确失败或安全降级，不产生任何厂商直连。
- 两名成员的真实计量相互独立；每个成员超过预算时网关返回 429/拒绝，且不会继续产生可计费 completion。
- 缺失或错误身份/虚拟 key 被拒；日志、trace 与审计不泄露凭证。
