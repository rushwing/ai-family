## 前置条件

- PG16、受限应用 role 与两名家庭成员（A、B）已就绪；应用 role 无 BYPASSRLS。
- 提供含目标、计划、打卡、报表与成员关系的 MariaDB 脱敏 fixture；迁移脚本与 Alembic 基线可重复执行。
- agent-core 创建 checkpoint、outbox、audit、confirm_token、tool-call log 等 tenant-bearing 平台表。

## 步骤

1. 对 fixture 执行一次 MariaDB→PG16 数据迁移，再重复执行迁移。
2. 以 A、B 的 per-transaction member claim 分别读写业务表；分别尝试读取对方数据和把记录写给对方。
3. 对 checkpoint、outbox、audit、confirm_token、tool-call log 重复步骤 2。
4. 检查所有迁移创建的 tenant-bearing 表均有 family_member_id（或明确的 shared scope）、ENABLE + FORCE RLS、读写策略及受限 role 授权。
5. 运行 RLS 扫描；再以临时的无 RLS 新表和裸 SECURITY DEFINER 迁移作为负例运行扫描。

## 期望结果

- 首次迁移保留业务语义与关联；再次执行不产生重复业务记录或破坏 Alembic 版本。
- A/B 在 API 与数据库层均只能看到和写入自己的数据；跨成员读为 0 行、跨成员写被拒。
- 平台状态表不成为租户隔离旁路，结果与业务表一致。
- 每张受管表均强制 RLS；应用 role 无 BYPASSRLS；正常扫描通过，两个负例均失败。
