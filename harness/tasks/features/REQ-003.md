## 背景

goal-agent 现状：FastAPI + FastMCP（36 tools）+ MariaDB + Telegram + APScheduler，单用户角色模型
（X-Telegram-Chat-Id header 鉴权）。需迁移到平台基座并 LangGraph 化，成为第一个垂直切片。

## 需求描述（差距清单；scope 重审与 L2 架构见 docs/design/08，现状盘点见 06）

> 经 goal-agent scope 重审（docs/design/08）：M1 = 核心目标闭环 + 周/月报 + dogfooding + GoalGroup 向导（精简）。
> 取舍/改造与 defer 项见下；BUG-001/002/011 升为**设计内建需求**（非兜底验收）。

1. **数据迁移（含 RLS 内建）**：MariaDB → PostgreSQL 16（SQLAlchemy 驱动切换 + Alembic 基线重建 + 数据搬迁脚本），
   所有业务表加 `family_member_id` 并启用 RLS；**RLS 加固随迁移一次性内建**（FORCE RLS、迁移受限 role、worker/图节点必设 member/shared scope、默认禁 SECURITY DEFINER、PgBouncer 事务内 SET LOCAL、CI 扫新表 RLS —— BUG-002 critical）；**平台状态表**（langgraph checkpoint/outbox/audit/confirm_token/tool log）同为 tenant-bearing，一并纳入 RLS + CI 扫描（BUG-014）
2. **身份改造**：Header 鉴权 → OIDC JWT（IdP 签发），best_pal→admin / go_getter→adult·kid；MCP tools 改为工具侧鉴权
3. **编排改造（含幂等 + 状态权威）**：重构为 LangGraph 长程图（goal → milestone → daily plan → check-in 循环）+ GoalGroup 精简 wizard 子图；PG checkpointer 持久化，APScheduler 触发改为图唤醒入口；**副作用节点的业务写入+outbox+idempotency_key 同一事务、checkpoint 仅记控制流、恢复以 DB 业务状态为权威 reconcile**（BUG-001/BUG-013）
4. **保留资产 + 工具 manifest**：36 个 MCP tools 语义不变接入 MCP 网关，**每个 write 类 tool 补 risk 元数据 + 工具侧 JWT，无 auth/无测试者拒绝注册**；逐 tool manifest（risk/角色/tenant scope/两段式/测试）；**M1 禁旧 REST/OpenClaw 直连、只走网关**（ADR-010/REQ-004/BUG-017）；streak/XP/报表逻辑保留
5. **外部通道改造**：GitHub 归档 → MinIO + PG 版本表（数据不出家门 P4）；Telegram → `notify.out` 统一出口（ADR-006）
6. **审计内建**：关键事件写**独立 audit schema**（append-only，**INSERT-only role + 禁 UPDATE/DELETE**，与可删会话历史物理分离，kid 不可删 —— BUG-011/BUG-015）
7. **kid 结构化只路径（M1）**：kid 仅 RLS 只读自有目标 + Draft-First 打卡 + 模板化赞语（不走 LLM 自由生成）；**不可自由对话、不可创建/批准平台或他人目标写入**，越界转家长（BUG-012）
8. **ChatUI v0**：NextJS 自研，登录 + 对话 + 目标看板三屏；**scope 锚点 = Claude Design 高保真原型 `docs/design/prototypes/chatui-v0.html`**（计划确认/Draft-First confirm token/kid 引导态 + 角色感知，pending 08 评审 delta）
9. **Dogfooding**：将本仓库 M0–M5 里程碑与 REQ 列表录入为第一个 Goal

## 非目标

- 不接 IntentRouter/Planner（M2）；不做多语言。"计划确认"是 GoalAgent 内部 `interrupt()` 确认门，**非 M2 Planner 的 Plan Mode**（BUG-018）
- **kid 自由对话 + 完整 Compliance v0**（输入注入/意图过滤 + 输出 kid 100% judge + PII 脱敏 + compliance_block 审计）顺延 M3；M1 kid 走结构化只路径（见需求 7 / BUG-012）
- **scope 重审 defer 项（均带复活触发，见 08 §2）**：web_research + feasibility（外部数据→BUG-005 消毒中间件就绪后）、Gamification+（排行榜/streak freeze/XP shop，核心闭环上线后增量）、OpenClaw 插件通道（M4 再评估复活）；wizard 的 research/feasibility 节点 M1 移除

## 验收标准明细

1. 两个不同成员账号各建一个目标，互相在 API 与 DB 层面均不可见对方数据
2. kill 掉 Agent 进程后重启，进行中的月度计划任务从 checkpoint 恢复并继续
3. Langfuse 中可检索到从 ChatUI 请求到工具调用的完整 trace
4. GoalAgent 产出的首份周报包含 ai-family 项目真实进度
5. **幂等（BUG-001）**：在 `daily_plan` 下发 / `evaluate` 计分 / 提醒发送之后 kill 进程并重启续跑，**不产生重复写库、重复通知或重复计分**（同一 idempotency_key 二次执行被去重）
6. **审计 append-only（BUG-011）**：以 app 角色及 kid 身份对审计表执行 `UPDATE`/`DELETE` 均被拒绝（仅 append）；kid 无法删除/软删自己的操作记录
7. **RLS 绕过封堵（BUG-002）**：除常规查询外，经迁移/DDL role、后台 worker/图节点、连接池复用（PgBouncer 事务模式）、SECURITY DEFINER 等路径，跨成员读取仍返回 0 行；CI 对无 RLS 策略的新表判失败
