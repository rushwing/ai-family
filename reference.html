System Design: Enterprise Multi-Agent Assistant

An enterprise AI platform for a large financial institution — enabling knowledge Q&A, document analysis, data queries, report generation, approval workflows, and customer communication, with full data governance.

Enterprise AI Platform Multi-Agent Workflow RAG + Tool-Calling Data Governance / DLP RBAC / ABAC Explainability / Audit

Problem Scope (as given)

A large financial client wants an enterprise GenAI Assistant for employees: knowledge Q&A, document analysis, DB queries, report generation, approval flows, email drafts, meeting summaries. Must connect to internal knowledge bases, databases, CRM, ticketing, and document systems. Must meet data security, permission isolation, explainability, and audit requirements.

Step 1 — Clarify Before Drawing Anything

1. Which business scenarios have highest priority? Knowledge Q&A? Customer service? Sales enablement? Financial analysis? Compliance review? Start vertical, then generalize.
2. Who are the users and how many? Frontline staff, relationship managers, legal, risk, management? 100 users or 100,000?
3. What actions is the agent allowed to take? Read-only? Generate drafts only? Submit approvals? Execute DB updates?
4. What data sources exist? SharePoint / Google Drive / Confluence / CRM / BigQuery / Ticketing?
5. What is the permission model? Must it inherit original system permissions? Row-level / document-level security required?
6. What compliance requirements exist? Data residency? PII masking? SOC2 audit trail? Human approval for write actions?
7. What is the rollout target? One high-value workflow first, or a reusable platform?
🎯 Key FDE answer: "I would not start by building a universal enterprise assistant. I'd select one high-value, low-risk, measurable workflow — e.g. 'relationship manager auto-generates client briefing' or 'compliance team reviews contract clauses.' First build a vertical agent, then extract a reusable platform capability."

Architecture Diagram

Agent Runtime Design

Intent Router   → classifies intent, estimates risk level (read / draft / write / high-risk)
Planner         → decomposes task, selects agents, determines if human approval needed
Knowledge Agent → document retrieval, citation, summarization
Data Agent      → generates constrained SQL, queries warehouse (read-only by default)
Compliance Agent→ checks output for policy violations, PII, sensitive terms
Workflow Agent  → interacts with CRM / ticket / approval systems (draft-first mode)
Response Synth. → merges evidence, produces final answer with citations + action summary

Draft-First Execution Model

For all write actions (send email, submit approval, update CRM): the agent drafts first, the user reviews and confirms. The agent cannot auto-execute write actions without explicit user approval. This is non-negotiable for enterprise financial contexts.

Context Engineering

Prompt context =
  system_policy        # enterprise rules, tone, scope
  + user_role          # from identity provider
  + task_intent        # classified by intent router
  + allowed_tools      # scoped to user role
  + retrieved_evidence # from knowledge/data agents
  + conversation_state # session memory
  + previous_tool_results
  + risk_level
  + output_schema      # structured output for UI + audit
⚠️ Business rules should NOT live entirely in the prompt. Route them to a policy engine. Prompts that carry all rules become unmaintainable and untestable.

Security Controls

Identity & Access

  • Federate identity from enterprise IdP (OIDC/SAML)
  • RBAC for agent capabilities; ABAC for data rows/documents
  • Tool calls carry user identity — tools enforce authz, not the agent
  • Document-level permission check before returning any content

Data Safety

  • PII detection and redaction before LLM context injection
  • DLP policy prevents exfiltration via tool output
  • Data residency controls (regional deployment)

Prompt Safety

  • Prompt injection detection on user-uploaded documents
  • Input sanitization before system prompt composition
  • SQL Agent: whitelist-only query patterns, read-only DB user

Audit

  • Every prompt, tool call, retrieved doc, and output logged with trace_id and user_id
  • Immutable audit log for regulatory inspection
  • Anomaly detection on unusual access patterns
🎯 Key: "An agent gaining permission by reading context is not the same as the user having that permission." Tools enforce authz independently.

Evaluation & Rollout Strategy

Offline Metrics

  • Q&A accuracy on golden dataset
  • Retrieval precision/recall
  • Tool call correctness (right tool, right params)
  • Hallucination rate (unsupported claims)
  • PII leak rate in outputs

Online Metrics

  • Task completion rate
  • Draft adoption rate (user sends AI-generated email without major edit)
  • Time-to-answer vs. baseline
  • Human approval rejection rate (high = agent too aggressive)

Rollout Playbook

  1. Pick one high-value vertical workflow
  2. Shadow mode: AI generates recommendations, humans execute; collect feedback
  3. Draft mode: AI generates drafts, humans review and confirm
  4. Assisted mode: AI executes low-risk actions autonomously
  5. Extract reusable platform components for next workflow

ROI Framing for C-Suite

  • Hours saved per user per week × headcount × cost
  • Reduction in compliance review cycle time
  • Customer response time improvement
  • Error rate reduction in manual workflows

Scalability

The PDF tests: "thinking through the jump from 10,000 internal users to millions of external clients." For an enterprise assistant this means multi-tenant scale with strict isolation.

10K → 1M Users: Architecture Shift

  • 10K users (single enterprise): Cloud Run for tools, Agent Engine for agent runtime, Vertex AI Search single corpus. Everything in one project/region.
  • 100K users (large enterprise / multi-BU): Add per-BU data isolation (separate Search corpora, BQ datasets). Quota management per team.
  • 1M+ users (multi-tenant SaaS): Tenant-per-namespace in Agent Engine. VPC SC perimeters per large tenant. Cloud Armor for DDoS. Global load balancing with regional failover.

Cloud Run Auto-scaling

  • Agent tools on Cloud Run scale from 0 to thousands of instances automatically.
  • Set min-instances=2 for P1 tools to avoid cold start latency spike.
  • CPU-based autoscaling for predictable load; request-count scaling for spiky traffic.

Data Isolation at Scale

  • Vertex AI Search: separate corpus per tenant or per department — no cross-tenant document bleed.
  • BigQuery: row-level security policy binding user_email → department → allowed_tables.
  • VPC Service Controls: perimeter per enterprise customer if running as managed service.
  • Agent session namespace by tenant_id — no shared state.

Quota & Rate Management

  • Per-user token budget enforced at the agent gateway layer.
  • Per-team daily/monthly spend quota via Cloud Billing alerts.
  • Vertex AI API quotas per project — request increases proactively before go-live.
  • Back-pressure: queue excess requests rather than dropping them; communicate wait time to user.

Performance & Cost Optimization

Model Routing by Task Complexity

  • Flash (~$0.075/M tokens): simple Q&A, summarization, routing decisions, email drafts.
  • Pro (~$3.50/M tokens): complex contract review, multi-document analysis, financial reasoning.
  • Intent Router classifies task complexity → routes to appropriate model tier automatically.
  • Estimated: 80% of requests use Flash → ~10× cost reduction vs. routing everything to Pro.

Prompt Caching

  • Gemini supports prefix caching — cache the system prompt + policy context (these don't change per request).
  • Saves ~40–60% of input tokens on repeated similar requests.
  • Cache TTL: 1 hour for session context; 24 hours for static policy documents.

Retrieval Cost Control

  • Start with top-k=3 retrieved documents; expand to k=8 only for complex multi-document tasks.
  • Cache Vertex AI Search results for common queries (e.g. "what is the expense policy" — same answer for all users).
  • Batch embedding generation for document ingestion — don't regenerate unchanged documents.

Async & Batch Processing

  • Meeting summaries, weekly reports, compliance audits: run as async batch jobs (Gemini Batch API), not real-time — ~50% cost discount.
  • Heavy document ingestion (SharePoint sync) runs on a schedule, not on user request.
  • Pre-compute common analytics summaries nightly in BigQuery — agent serves cached summaries for standard dashboards.

Latency

  • Target: p99 < 3s for simple Q&A; < 10s for complex analysis (set expectations with UX design).
  • Stream tokens to UI as soon as first token arrives — don't buffer full response.
  • Parallelize independent tool calls in planner (CRM fetch + doc search simultaneously).

Likely Follow-up Questions

  • How to prevent prompt injection? — Treat user-uploaded content as untrusted data, not instructions. Separate instruction and data channels. Use a content scanner before injecting into context.
  • User uploads malicious doc to override system instructions? — Strict delimiter separation between system prompt and retrieved content. Instruction override detection layer.
  • SQL Agent prevents sensitive field access? — DB user has read-only access on a view that excludes PII columns. Agent generates SQL against schema registry, not raw tables.
  • How do agents share state? — Shared session context object, passed through the planner. No direct agent-to-agent messaging. Planner owns state merging.
  • Fallback when agent fails? — Graceful degradation: surface partial results + "I couldn't complete X, here's what I found." Never hallucinate missing results.
  • Customer says "we want ChatGPT for everything" — how do you respond? — Acknowledge the goal. Redirect to: "Let's identify which workflows generate the most value. Trying to automate everything at once usually means doing nothing well. What would save your team the most time in the next 90 days?"
